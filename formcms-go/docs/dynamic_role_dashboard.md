# Design Specification: Dynamic Role-Based Dashboard & Navigation

## 1. Overview
The goal of this specification is to design and implement a dynamic, role-based home page/dashboard in the FormCMS Go system. When a user logs in, their default role will dictate the initial dashboard layout and navigation menu they see. If a user holds multiple roles, they can seamlessly switch between them using a UI dropdown, dynamically updating their dashboard and navigation to reflect the selected role's context.

## 2. Architectural Brainstorming: Where to Save the Dashboard?

**Objective:** Store the role-specific dashboards in a way that aligns with FormCMS's dynamic, headless architecture and allows modifications during runtime and deployment.

**Options Considered:**
1. **Static HTML/JS files in `wwwroot`:** Hard to modify at runtime without file system access. Does not align with the dynamic CMS nature.
2. **Hardcoded inside the Frontend framework:** Requires recompilation/redeployment for every dashboard change.
3. **Database-backed "Page" Entity (Chosen Approach):**
   * FormCMS already has a powerful `Page` entity and a built-in page designer (GrapesJS).
   * FormCMS also has a `Menu` entity schema for dynamic navigation.
   * **Conclusion:** The most logical, architecture-aligned approach is to link a `Role` entity directly to a `Page` entity (for the dashboard) and a `Menu` entity (for navigation). This utilizes the existing GrapesJS designer for runtime dashboard creation and allows deployment-time seeding via standard JSON data imports.

## 3. Design Specification

### 3.1. Data Model Enhancements
We will extend the existing dynamic schemas in `apps/rbac/schemas/`:

* **User Schema (`user.json`):**
  * Add `default_role_id` (Link to `Role`). This ensures the system knows which dashboard to show immediately upon login.
  
* **Role Schema (`role.json`):**
  * Add `dashboard_page_id` (String/Link to `Page`). Points to the GrapesJS page designed for this role.
  * Add `menu_id` (String/Link to `Menu`). Points to the navigation menu specific to this role.

### 3.2. Backend API Changes
* **`/api/auth/me` & `/api/auth/login`:**
  * Enhance the response to include the user's `default_role_id`, a list of all assigned `Roles`, and the `dashboard_page_id` and `menu_id` for each assigned role.
* **UI Context Context vs. Security Context:**
  * The backend JWT will still contain *all* authorized roles for the user (to prevent needing a new token every time the UI role is switched). The "Active Role" switch is purely a frontend context shift to change what the user *sees* (Dashboard and Menu), while backend RBAC continuously protects endpoints based on the union of their assigned roles.

### 3.3. Frontend & UI Workflows
* **Login Routing:**
  1. User authenticates.
  2. Frontend retrieves user profile and checks `default_role_id`.
  3. Frontend sets `localStorage.setItem('activeRoleId', default_role_id)`.
  4. Frontend redirects to the dynamic dashboard route.
* **Role Switcher Dropdown:**
  * Embedded in the global Navbar.
  * Visible only if `user.roles.length > 1`.
  * Changing the dropdown updates `localStorage.getItem('activeRoleId')` and triggers a page reload or dynamic re-render of the Dashboard and Menu.
* **Dynamic Navigation (`navbar.js`):**
  * Instead of hardcoded links, `navbar.js` will fetch the `Menu` entity associated with the `activeRoleId` and render the links accordingly.
* **Dynamic Dashboard:**
  * The home route (`/` or `/dashboard`) will fetch the `Page` entity associated with the `activeRoleId`.
  * **Fallback Mechanism:** If the selected role does not have a `dashboard_page_id` assigned (e.g., a newly created role), or if the assigned page is not found, the system will fall back to displaying a predefined "Default Dashboard" (e.g., a system-level default page or a hardcoded standard overview).
  * It will then render the HTML/CSS generated by GrapesJS into the DOM container.

### 3.4. Runtime & Deployment Management
* **Runtime Modification:** System Administrators can go to the Admin Panel -> Pages -> Create New Page (using GrapesJS). Then, go to Roles -> Edit Role -> Set `dashboard_page_id` to the newly created page.
* **Deployment Modification:** Administrators can export the Page, Menu, and Role records into JSON files (e.g., `apps/rbac/data/test_data.json`) which the system automatically ingests and provisions upon deployment.

## 4. Implementation Plan

### Phase 1: Schema Updates
- [ ] **Task 1.1:** Update `apps/rbac/schemas/role.json` to include `dashboard_page_id` and `menu_id` attributes.
- [ ] **Task 1.2:** Update `apps/rbac/schemas/user.json` to include `default_role_id`.
- [ ] **Task 1.3:** Create or update initialization data (`apps/rbac/data/test_data.json`) to map the `admin` role to an `admin_dashboard` page and `admin_menu`.

### Phase 2: Backend API Enhancements
- [ ] **Task 2.1:** Modify `AuthService.ValidateToken` or create a new `GetMe()` method to deeply fetch the user's roles, including the `dashboard_page_id` and `menu_id` associated with those roles.
- [ ] **Task 2.2:** Update the `/api/auth/me` endpoint to return this structured role data to the frontend.

### Phase 3: Frontend Navigation & State Management
- [ ] **Task 3.1:** Update `js/utils/user.js` to handle `activeRoleId` state (saving/loading from `localStorage`).
- [ ] **Task 3.2:** Modify `js/components/navbar.js` to build the navigation links dynamically by fetching the Menu entity defined by `user.activeRole.menu_id`.
- [ ] **Task 3.3:** Inject a `<select>` dropdown in the Navbar if the user has more than one role, wiring the `onChange` event to update `activeRoleId` and refresh the UI.

### Phase 4: Dynamic Dashboard Rendering
- [ ] **Task 4.1:** Create a new `dashboard.html` (or modify `index.html`) to act as the landing page.
- [ ] **Task 4.2:** On load, the dashboard fetches `user.activeRole.dashboard_page_id`.
- [ ] **Task 4.3:** Implement the fallback logic: If `dashboard_page_id` is null or the fetch fails, load the system-defined `default_dashboard` page.
- [ ] **Task 4.4:** Fetch the page content via the existing `/api/pages/{id}` endpoint and render the HTML/CSS components directly into the DOM container.

### Phase 5: Testing & Verification
- [ ] **Task 5.1:** Create a "System Admin" role and a standard "User" role in the database.
- [ ] **Task 5.2:** Assign both roles to a test user.
- [ ] **Task 5.3:** Create two separate Pages via the GrapesJS editor. Link them to the respective roles.
- [ ] **Task 5.4:** Log in as the test user. Verify the default dashboard loads.
- [ ] **Task 5.5:** Use the dropdown to switch roles. Verify the navigation menu and dashboard content change seamlessly without requiring re-authentication.
